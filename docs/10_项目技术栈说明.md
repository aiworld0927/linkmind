## 一、技术栈总览

LinkMind 知识图谱系统采用现代化的前后端分离架构，结合图数据库技术，构建高性能的知识可视化平台。

## 二、后端技术栈详解

### 2.1 FastAPI

**核心特性**：
- 基于 Starlette 和 Pydantic，支持异步请求处理
- 自动生成 OpenAPI 3.0 规范文档（Swagger UI）
- 类型提示驱动的请求参数校验
- 依赖注入系统
- 中间件支持

**项目应用**：
- 路由定义：使用 `@app.get/post/put/delete` 装饰器
- 请求模型：使用 Pydantic BaseModel 定义
- 统一响应：通过 `success_response` 和 `error_response` 工具函数
- CORS 配置：允许前端跨域访问
- 异常处理：全局异常捕获中间件

**关键文件**：
- [main.py](file:///coding/linkmind/backend/src/main.py) - 应用入口，路由注册
- [utils/response.py](file:///coding/linkmind/backend/src/utils/response.py) - 统一响应格式
- [middleware/exception_handler.py](file:///coding/linkmind/backend/src/middleware/exception_handler.py) - 全局异常处理

### 2.2 Neo4j

**核心特性**：
- 原生图数据库，存储节点和关系
- Cypher 查询语言，声明式图查询
- 支持社区检测算法（Louvain）
- 全文索引支持
- ACID 事务保证

**项目应用**：
- Note 节点存储知识条目
- LINK 关系存储节点间依赖
- 全文索引加速搜索
- 社区聚类实现知识分组

**关键文件**：
- [database/neo4j_connection.py](file:///coding/linkmind/backend/src/database/neo4j_connection.py) - Neo4j连接管理
- [database/init_indexes.py](file:///coding/linkmind/backend/src/database/init_indexes.py) - 索引初始化
- [models/note.py](file:///coding/linkmind/backend/src/models/note.py) - 笔记模型
- [models/link.py](file:///coding/linkmind/backend/src/models/link.py) - 关系模型

### 2.3 后端架构模式

采用经典的三层架构：

```
┌─────────────────────────────────────┐
│           API 路由层 (Routes)        │
│  处理HTTP请求，参数校验，调用Service  │
├─────────────────────────────────────┤
│           业务服务层 (Services)       │
│  业务逻辑处理，事务管理，数据转换      │
├─────────────────────────────────────┤
│         数据访问层 (Repositories)     │
│  Cypher查询，数据库操作封装           │
├─────────────────────────────────────┤
│              Neo4j 数据库            │
└─────────────────────────────────────┘
```

## 五、技术选型决策记录

### 5.1 为什么选择 FastAPI 而非 Django/Flask？

| 对比维度 | FastAPI | Django | Flask |
|----------|---------|--------|-------|
| 性能 | 高（异步） | 中 | 低 |
| API文档 | 自动生成 | 需插件 | 需插件 |
| 类型提示 | 原生支持 | 需插件 | 需插件 |
| 学习曲线 | 低 | 高 | 中 |

**决策**：FastAPI 提供更好的开发体验和性能，适合快速构建 RESTful API。

### 5.2 为什么选择 Neo4j 而非关系型数据库？

| 对比维度 | Neo4j | MySQL/PostgreSQL |
|----------|-------|------------------|
| 关系查询 | 原生支持，高效 | JOIN操作，复杂关系性能差 |
| 社区检测 | 内置算法 | 需复杂查询 |
| 数据模型 | 图结构，灵活 | 表结构，固定Schema |
| 路径查询 | 高效 | 复杂且慢 |

**决策**：知识图谱场景需要频繁的关系查询和路径分析，Neo4j 是最佳选择。

### 5.3 为什么选择 Zustand 而非 Redux？

| 对比维度 | Zustand | Redux |
|----------|---------|-------|
| 代码量 | 轻量（~1KB） | 重（需配置） |
| Provider | 不需要 | 必须 |
| 学习曲线 | 低 | 高 |
| TypeScript | 原生支持 | 需额外配置 |

**决策**：Zustand 更简洁，适合中小型项目，无需复杂的配置。

### 5.4 为什么选择 X6 而非 D3.js？

| 对比维度 | X6 | D3.js |
|----------|-----|-------|
| 图编辑能力 | 内置拖拽、连线、对齐 | 需自行实现 |
| 节点渲染 | 支持React组件 | SVG操作复杂 |
| 事件系统 | 完善 | 需自行管理 |
| 学习曲线 | 低 | 高 |

**决策**：X6 提供开箱即用的图编辑能力，开发效率更高。
